# msgvault Docs Integration Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Move the public `msgvault` docs from `~/code/msgvault-docs` into this repo as a Zensical site rooted at `docs/`, with media hydrated from `docs-assets` and `docs-generated-assets` orphan branches.

**Architecture:** Port source docs and generation inputs into `docs/`, while keeping generated/static media out of main. Zensical, Vercel, hydration scripts, and check scripts mirror the shipped `agentsview` / `roborev` pattern, with msgvault-specific asset inventories and screenshot generation preserved.

**Tech Stack:** Go/testify for docs tooling tests, Bash/Git orphan refs for asset branches, uv + Zensical, Python validation scripts, Vercel, Docker/tmux/freeze screenshot generation, Chrome/ImageMagick diagram rendering.

- [ ] **Step 5: Add asset scripts**

Adapt `agentsview` scripts with `MSGVAULT_` environment variable names:

- `MSGVAULT_DOCS_ASSETS_BRANCH`
- `MSGVAULT_DOCS_GENERATED_ASSETS_BRANCH`
- `MSGVAULT_DOCS_USE_LOCAL_ASSET_BRANCHES`
- `MSGVAULT_DOCS_STATIC_ASSETS_DIR`

Expected asset arrays must exactly match this plan's Asset Inventory.

`docs/assets/hydrate-assets.sh` must:

- default to fetching `origin/docs-assets` and `origin/docs-generated-assets`, which is what Vercel needs;
- honor `MSGVAULT_DOCS_USE_LOCAL_ASSET_BRANCHES=1` by reading local `docs-assets` and `docs-generated-assets` refs before remote fetches, which is required for first-time pre-push validation;
- fail if either branch is unavailable or hydrated assets are missing expected files.

Publisher scripts must:

- reject symlinks;
- reject unexpected files, including dotfiles except `.DS_Store`;
- copy only expected assets;
- update local branch refs from a temp git repo without checking out/switching branches;
- support `--source` and `--push`.

Additional generated-asset publisher requirements:

- `docs/screenshots/update-generated-assets-branch.sh` default mode must regenerate assets by running both `docs/screenshots/generate-all.sh` and `docs/diagrams/build.sh` before validating and publishing.
- `--source DIR` publishes an already generated directory without running generators. Use this mode only after `bash docs/screenshots/generate-all.sh` and `bash docs/diagrams/build.sh` have already succeeded in the same workflow.
- If a `--skip-generate` mode is retained from a reference script, document it in the script help and cover it in `scripts/docs_assets_test.go`.
- The generated publisher must fail if the generated source directory is missing, empty, or lacks any expected asset; it must never publish an empty branch after stale generated assets are removed.
