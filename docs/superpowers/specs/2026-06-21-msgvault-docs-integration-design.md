# msgvault Docs Integration Design

## Goal

Move the public `msgvault` documentation from `~/code/msgvault-docs` into this
repository on the `integrate-docs` branch, port the site to Zensical like
`roborev` and `agentsview`, and keep screenshots and generated media out of the
main branch by hydrating them from orphan asset branches.

This is a porting project. The migration should preserve the current
`msgvault-docs` content, generation workflows, and published URL shape unless a
change is required by the Zensical port or by the asset-branch layout.

## Requirements

- Public docs source lives under `docs/` in this repository.
- The published site remains rooted at `https://msgvault.io`.
- Vercel treats `docs/` as the project root and uses docs-local Zensical,
  Python, and Vercel configuration.
- Astro/Starlight MDX pages from `~/code/msgvault-docs/src/content/docs/` are
  ported to Zensical-compatible Markdown.
- The public navigation from `msgvault-docs/astro.config.mjs` is preserved in
  `docs/zensical.toml`.
- Image media is not committed to the main branch under `docs/`.
- Asset publication uses two orphan branches:
  - `docs-assets` for curated static assets such as favicons, Open Graph media,
    hand-authored diagrams, and static SVGs.
  - `docs-generated-assets` for generated screenshots and rendered concept
    images.
- Local and Vercel builds hydrate assets before strict Zensical builds.
- Existing repo-internal docs that remain useful are preserved under
  `docs/internal/` and excluded from the published site.
- The current screenshot and demo-data generation approach from
  `msgvault-docs` is preserved. Demo data continues to be regenerated by the
  existing generator; do not introduce a fixture branch or change the screenshot
  generation architecture.

## Source Material

The migration has three inputs:

1. `~/code/msgvault-docs`
   - Public Astro/Starlight docs, MDX content, custom components, styles,
     current media, `vhs/` screenshot generation scripts, and `diagrams/`
     concept-diagram render scripts.
2. `~/code/agentsview` commit `1df98ca7`
   - The closest example for integrating a separate docs project into a parent
     repository with Zensical, Vercel rooted at `docs/`, asset hydration, and
     generated-asset orphan branches.
3. `~/code/roborev`
   - The Zensical docs layout and deployment pattern to match for docs-local
     config, Vercel commands, build wrappers, and validation scripts.

The public Markdown files should live directly under `docs/` and its public
subdirectories, not under `docs/docs/`. The wrapper script `zensical-docs.sh`
should build from a temporary public-docs copy so maintainer-only files such as
`README.md`, `internal/`, `screenshots/`, scripts, and superpowers specs are not
published.

## Zensical Port

The content port should keep page intent, headings, examples, and URLs stable
while translating Astro/Starlight-specific syntax to Zensical-compatible
Markdown:

- Convert `.mdx` files to `.md`.
- Replace Starlight `CardGrid` and `Card` usage on the home page with plain
  Markdown or HTML that Zensical renders cleanly.
- Replace Starlight `Aside` components and `:::note` / `::::tip` style blocks
  with supported Python-Markdown admonitions such as `!!! note`, `!!! tip`,
  `!!! caution`, and `!!! danger`.
- Replace the Astro `Screenshot.astro` component with ordinary `<figure>` /
  `<img>` markup using the hydrated asset paths.
- Preserve custom lightbox behavior by moving `public/lightbox.js` to
  `docs/javascripts/lightbox.js` and wiring it through `extra_javascript`.
- Preserve the monochrome msgvault docs styling by porting
  `src/styles/custom.css` to `docs/stylesheets/extra.css`, adjusted only as
  needed for Zensical's generated markup.
- Preserve install script URLs (`/install.sh`, `/install.ps1`) through Vercel
  redirects to the repository's raw scripts.

## Asset Ownership

Main branch source should reference only hydrated asset paths:

- `/assets/static/...` for curated static media.
- `/assets/generated/...` for generated screenshots and generated concept
  renders.

Before moving files, create an implementation inventory of source pages, source
media, generated outputs, and generator inputs from `msgvault-docs`. Use that
inventory to drive the expected asset arrays in the hydration and publisher
scripts, and to catch missing referenced assets before the Zensical build.

The initial expected asset sets should be derived from `msgvault-docs/public/`.
The current source references include:

- Static assets:
  - `favicon.svg`, `favicon-192.png`, `favicon-512.png`
  - `og-image.png`, `og-image.svg`
  - `how-it-works.svg`
  - `oauth-multi-account.svg`
  - any other hand-authored static media retained from `public/`
- Generated screenshot-style assets:
  - `stats.svg`
  - `list-senders.svg`
  - `tui-*.svg`
- Generated concept assets:
  - `concepts/account-collection-concept.png`
  - `concepts/deduplication-concept.png`
  - `concepts/oauth-multi-account-concept.png`
  - `concepts/safety-ladder-concept.png`
  - `concepts/survivor-selection-concept.png`

During inventory, the source docs referenced `tui-filter-modal.svg`, but that
file was not present under `msgvault-docs/public/`. It is generated by the
existing `vhs/generate-screenshots.sh` workflow and should be regenerated with
that workflow before seeding `docs-generated-assets`.

## Screenshot And Diagram Generation

Keep the existing `msgvault-docs` generation model:

- `generate_demo_data.py` regenerates the demo database and analytics data.
- Docker builds the current `msgvault` binary from the product repository.
- The existing tmux/freeze script captures SVG TUI screenshots.
- Diagram HTML files remain source-controlled and render to PNG via the
  existing Chrome/ImageMagick script.

The only changes should be repository-path adaptation:

- Move `vhs/` contents into `docs/screenshots/` or the closest equivalent that
  matches the `roborev` / `agentsview` docs layout.
- Change default output paths from `public/` to ignored
  `docs/assets/generated/`.
- Publish generated outputs to `docs-generated-assets`.
- Keep generated demo data out of main branch history, as it is regenerated by
  the current workflow.

Do not add a third fixture branch, replace the Docker/freeze workflow, or alter
the data-generation strategy unless the existing workflow cannot run in this
repository without a small path fix.

## Existing In-Repo Docs

The current `msgvault/docs/` directory contains implementation-adjacent
material rather than the public website. Preserve durable references under
`docs/internal/` and exclude them from published output:

- `PG_STATUS.md`
- `recovery.md`
- `accounts-identities-collections-dedup/spec.md`
- useful prose from `accounts-identities-collections-dedup/README.md`
- PostgreSQL planning/reference notes that remain useful to maintainers

Do not publish old planning screenshots or duplicate public docs pages from
these files. The public account, identity, collection, deduplication,
PostgreSQL, recovery, and troubleshooting docs should come from the ported
`msgvault-docs` content.

## Build And Deployment

Mirror the `agentsview` / `roborev` pattern:

- `docs/pyproject.toml` and `docs/uv.lock` own the Zensical dependency set.
- `docs/vercel.json` uses:
  - `framework: null`
  - `installCommand: uv sync --frozen --no-dev`
  - `buildCommand: uv run --frozen bash ./vercel-build.sh`
  - `outputDirectory: site`
  - `trailingSlash: true`
- `docs/vercel-build.sh` hydrates assets and invokes the Zensical wrapper.
- Root `Makefile` gains docs targets:
  - `docs-install`
  - `docs-build`
  - `docs-serve`
  - `docs-check`
  - `docs-screenshots`
  - `docs-assets-branch`
  - `docs-generated-assets-branch`
  - `docs-deploy-staging`
  - `docs-deploy`
- `scripts/update-docs.sh` provides the manual deploy workflow:
  regenerate generated assets, push generated asset branch, hydrate assets,
  build, check, then deploy.
- `.gitignore` covers docs-local build and hydrated output, including
  `docs/assets/static/`, `docs/assets/generated/`, regenerated demo data,
  temporary Zensical public-doc directories, and `docs/site/`.

## Validation

Add a root docs checker similar to `agentsview/scripts/check-docs.sh`:

- Reject root-level `zensical.toml` and `vercel.json`.
- Reject tracked docs image media on main.
- Reject tracked files under hydrated `docs/assets/static/` and
  `docs/assets/generated/`.
- Reject stale media references that do not use `/assets/static/` or
  `/assets/generated/`.
- Validate Markdown source shape, including YAML frontmatter and admonition
  formatting.
- Hydrate both asset branches.
- Run a strict Zensical build.
- Validate built-site expectations such as Open Graph media, install-script
  redirects, and required generated assets.

Add Go tests under `scripts/` using `github.com/stretchr/testify` to cover the
asset scripts:

- `hydrate-assets.sh` force-fetches remote asset branches and replaces stale
  hydrated local assets.
- Static and generated branch publisher scripts reject unexpected files and
  symlinks.

All Go test invocations must use the repository build tags:

```bash
go test -tags "fts5 sqlite_vec" ./scripts
```

## Non-Goals

- Do not preserve the Git history of `msgvault-docs` inside this repository.
- Do not commit generated screenshots, concept PNGs, or generated demo data to
  main.
- Do not redesign the screenshot generation workflow.
- Do not add a generated-fixtures branch.
- Do not redesign the docs information architecture beyond the minimal work
  required to port Astro/Starlight pages to Zensical.
- Do not change product code behavior as part of the docs migration.

## Success Criteria

- `docs/` is a self-contained Zensical/Vercel docs project.
- The public docs build with hydrated assets and no tracked docs media.
- The current screenshot and diagram generators still regenerate the same
  classes of assets from source.
- `docs-assets` and `docs-generated-assets` are seeded locally and ready to
  push.
- `make docs-check` passes.
- Relevant Go tests for docs asset tooling pass with `-tags "fts5 sqlite_vec"`.
